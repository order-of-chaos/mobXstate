# 06. IDE Plugins

## Цель

IDE-плагины должны дать один и тот же MobXstate workflow в разных редакторах:

- найти `createMachine(...)`;
- открыть viewer/simulator;
- открыть visual editor;
- показать diagnostics;
- сгенерировать typegen;
- безопасно предложить source patch.

Плагины не должны содержать собственные реализации analyzer, editor или type
compiler. Они должны подключать shared core.

## Общая архитектура плагинов

```text
IDE extension shell
  -> file discovery
  -> source ranges
  -> command registration
  -> diagnostics UI
  -> webview/browser host when available
  -> shared Node worker / CLI bridge
        -> devtools-core
        -> devtools-source
        -> devtools-typegen
        -> devtools-ui bundle
```

IDE shell не хранит business state editor. Он хранит только:

- active editor/document identity;
- webview/tool-window lifecycle;
- subscriptions/disposables;
- native diagnostics handles;
- native source edit application.

Worker/server хранит `SourceDocumentCache`, displayed machine pointer и
semantic source state.

## VS Code plugin

VS Code - первый IDE target, потому что его Webview API подходит для сложного
custom UI.

### Команды

- `MobXstate: Open Viewer`
- `MobXstate: Open Visual Editor`
- `MobXstate: Generate Typegen`
- `MobXstate: Check Current File`
- `MobXstate: Export Machine Config`

### Возможности редактора

- code lens above `createMachine(...)`;
- diagnostics for machine config;
- code actions for missing store members;
- source patch preview;
- generated typegen file;
- webview panel with viewer/editor.

### Webview messaging

Сообщения от extension к UI:

- `LOAD_MACHINE`;
- `UPDATE_RUNTIME`;
- `SHOW_DIAGNOSTICS`;
- `APPLY_THEME`;
- `SOURCE_PATCH_RESULT`.

Сообщения от UI к extension:

- `REQUEST_SOURCE`;
- `REQUEST_TYPEGEN`;
- `REQUEST_PATCH_PREVIEW`;
- `SEND_SEMANTIC_EDITS`;
- `APPLY_PATCH`;
- `COPY_EXPORT`;
- `SEND_RUNTIME_EVENT`, only when runtime bridge exists.

Сообщения от worker к UI:

- `DISPLAYED_MACHINE_UPDATED`;
- `EXTRACTION_ERROR`;
- `DISPLAYED_MACHINE_MISSING`;
- `TYPEGEN_UPDATED`;
- `SOURCE_CONFLICT`;

### Source patch safety

Every patch must:

- be generated by `devtools-source`;
- show diff preview;
- fail closed on unsupported AST;
- never silently rewrite unrelated code.

## WebStorm plugin

WebStorm - второй target. JetBrains IDE plugins can embed browser UI through
JCEF, so the same web bundle can be reused with a different host shell.

### Components

- IntelliJ action for `createMachine(...)`;
- Tool Window: MobXstate Devtools;
- JCEF browser for viewer/editor UI;
- Kotlin/Java plugin shell;
- Node worker or CLI bridge for analyzer/typegen;
- IntelliJ inspections for diagnostics;
- write command action for accepted source patches.

### Required behavior

- The plugin finds machine configs in TypeScript files.
- The Tool Window opens the selected machine.
- Diagnostics are linked to source ranges.
- Source patches use preview and IDE write action.
- The web bundle is shared with VS Code.

### Risks

- JCEF availability can vary by JetBrains runtime.
- TypeScript PSI and external TypeScript compiler may disagree on edge cases.
- Plugin packaging and signing are separate release work.

## Zed integration

Zed - третий target. The first Zed milestone should not assume a stable custom
visual panel API. Current public extension docs emphasize extension
capabilities such as language extensions, LSP integration, process access,
settings and MCP servers. Visual UI should be treated as a later capability or
implemented through external preview until the API is stable.

### Milestone A: diagnostics/typegen

- Zed extension starts MobXstate language worker or CLI.
- It reports diagnostics for `createMachine(...)`.
- It can generate/check typegen.
- It exposes commands for export/check.

### Milestone B: external preview

- Command starts local preview server.
- Browser opens viewer/editor outside Zed.
- Preview server reads selected file via worker protocol.
- Source patches are exported as text or patch file first.

### Milestone C: native visual UI

Only after Zed exposes stable visual panel/custom view APIs:

- embed `devtools-ui` bundle;
- support viewer/editor inside Zed;
- support patch preview inside editor.

## Межредакторный протокол

Use a small JSON-RPC-like protocol between host shell and Node worker.

Requests:

```text
analyzeFile(uri, text)
getMachine(uri, machineId)
compileTypegen(uri, machineId)
createPatch(uri, machineId, semanticEdits)
applyAcceptedPatch(uri, patchId)
formatExport(machineId, draft)
getNodePosition(uri, machineId, path)
getStoreBindingPosition(uri, machineId, binding)
```

Responses:

```text
MachineAnalysis
MachineSource
TypegenResult
PatchPlan
FormattedExport
Diagnostics
SourceConflict
```

The protocol must be versioned:

```json
{
  "protocol": "mobxstate-devtools.v1",
  "id": "request-id",
  "method": "analyzeFile",
  "params": {}
}
```

## Release order

1. Standalone devtools UI.
2. CLI analyzer/typegen.
3. VS Code extension.
4. WebStorm plugin.
5. Zed diagnostics/typegen.
6. Zed visual integration if API allows.
